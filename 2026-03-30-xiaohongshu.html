📕 小红书 MCP Skill

用命令行驾驭小红书 — 搜索、发布、互动、数据分析一站搞定

这个 Skill 是什么？

一句话：把小红书变成一个可编程的内容平台——搜索、读帖、发帖、评论、追热点，全部用命令行搞定。

这不是一个"教 AI 怎么写小红书"的 Prompt Skill。这是一套完整的工程化工具集：15 个 Shell/Python 脚本，通过 MCP（Model Context Protocol）协议连接小红书的底层能力，让 AI 助手可以直接操作你的小红书账号——搜索内容、获取帖子详情和评论、发布图文/视频、追踪热点话题并自动生成分析报告。

为什么值得关注？

小红书是目前中文互联网最有"内容价值"的平台之一——用户质量高、内容消费意愿强、种草转化率碾压其他平台。但小红书的运营全靠手动：搜关键词、看竞品、分析评论、定选题、写内容、发帖……每一步都是人肉操作。

这个 Skill 做的事，是把这条链路的关键节点都"API 化"了。虽然它目前还不能替你写出爆款笔记（那是内容 Skill 的事），但它能让你：

  • 核心 数据驱动选题——不用刷 App，直接拉取某个话题下的热帖和评论数据
  • 实用 自动化发布——AI 生成的内容可以直接发布到小红书，不用手动复制粘贴
  • 实用 竞品监控——追踪特定用户或话题，定期生成分析报告
  • 加分 跨平台联动——热点报告可以一键导出到飞书文档
核心理念：MCP 协议 + Shell 脚本 = 万物可编程

架构设计哲学

这个 Skill 的设计思路很"Unix 哲学"——每个脚本只做一件事，通过 MCP 协议统一调度。底层是一个叫 xiaohongshu-mcp 的 Go 服务（带 headless 浏览器），上层是一堆 Shell 脚本和 Python 工具。

核心数据流：feed_id + xsec_token

小红书的 API 有个重要机制：大多数操作需要 feed_id + xsec_token 配对。这对值只能从搜索/推荐/用户主页的结果中获取，不能自己构造。这意味着你必须先搜索，再操作——这是小红书的反爬设计，也是这个 Skill 需要理解的关键。

# 典型工作流：搜索 → 拿 feed_id → 获取详情 search.sh "咖啡" → 返回 feeds 数组，每个包含 id + xsecToken post-detail.sh "feed_id" "xsec_token" → 返回完整笔记：正文、图片、评论列表 # 拿到评论里的 comment_id 后才能回复 comment.sh "feed_id" "xsec_token" "写得真好！"

关键功能模块（可折叠）

1
🔍 内容搜索 — search_feeds
▼

支持关键词搜索 + 多维度筛选（排序、类型、时间范围、浏览状态、地理位置）。

# 基础搜索 search.sh "咖啡" # 带筛选条件的高级搜索 mcp-call.sh search_feeds '{ "keyword": "咖啡", "filters": { "sort_by": "最多点赞", "note_type": "图文", "publish_time": "一周内" } }'
筛选项可选值
sort_by综合 | 最新 | 最多点赞 | 最多评论 | 最多收藏
note_type不限 | 视频 | 图文
publish_time不限 | 一天内 | 一周内 | 半年内
search_scope不限 | 已看过 | 未看过 | 已关注
location不限 | 同城 | 附近
2
📖 帖子详情 — get_feed_detail
▼

获取完整笔记内容：正文、图片列表、评论列表。支持滚动加载全部评论、展开二级回复。

# 基础用法 post-detail.sh "feed_id" "xsec_token" # 加载全部评论（含二级回复） mcp-call.sh get_feed_detail '{ "feed_id": "...", "xsec_token": "...", "load_all_comments": true, "limit": 50, "click_more_replies": true, "scroll_speed": "fast" }'

⚠️ load_all_comments 会模拟浏览器滚动加载，数据量大时较慢。建议先用 false 预览前 10 条。

3
🔥 热点追踪 — track-topic
▼

这是整个 Skill 最有价值的功能模块。一行命令：搜索 → 批量拉取帖子详情 → 分析评论 → 生成 Markdown 报告。还能一键导出到飞书。

# 追踪 DeepSeek 话题，拉取 5 篇热帖 track-topic.sh "DeepSeek" --limit 5 # 输出报告到文件 track-topic.sh "春节旅游" --limit 10 --output report.md # 同时导出到飞书文档 track-topic.sh "iPhone 16" --limit 5 --feishu

报告包含：概览统计（总帖子/点赞/评论数）→ 热帖详情（作者、正文、互动数据、热门评论）→ 评论关键词 → 趋势分析（热度指数、互动活跃度）

底层由 track-topic.py（313行 Python）实现，调用 mcp-call.sh 完成数据采集。

4
💬 互动操作 — 评论/点赞/收藏
▼

支持完整的互动链路：发表评论、回复评论、点赞/取消点赞、收藏/取消收藏。

# 发表评论 comment.sh "feed_id" "xsec_token" "写得真好！" # 回复特定评论（需要 comment_id + user_id） mcp-call.sh reply_comment_in_feed '{ "feed_id": "...", "xsec_token": "...", "content": "谢谢分享！", "comment_id": "...", "user_id": "..." }' # 点赞（取消：加 "unlike": true） mcp-call.sh like_feed '{"feed_id":"...","xsec_token":"..."}' # 收藏 mcp-call.sh favorite_feed '{"feed_id":"...","xsec_token":"..."}'
5
📝 内容发布 — 图文 & 视频
▼

支持发布图文笔记和视频笔记，可带话题标签，支持定时发布。

# 发布图文笔记 mcp-call.sh publish_content '{ "title": "今日咖啡推荐", "content": "这家店的手冲超绝...", "images": ["/path/to/img1.jpg", "/path/to/img2.jpg"], "tags": ["咖啡", "探店"] }' # 定时发布（1小时~14天内） mcp-call.sh publish_content '{ "title": "...", "content": "...", "images": ["..."], "schedule_at": "2026-03-31T10:00:00+08:00" }' # 发布视频笔记 mcp-call.sh publish_with_video '{ "title": "Vlog", "content": "今天的记录", "video": "/path/to/video.mp4" }'

⚠️ 限制：标题 ≤ 20字符，正文 ≤ 1000字符，日发布 ≤ 50条，图片至少1张。

6
🖼️ 长图导出 — export-long-image
▼

把帖子内容（标题+正文+图片）拼接成白底黑字的 JPG 长图，类似小红书"分享截图"功能。支持自定义宽度和画质。

# 从 JSON 文件导出长图 python3 export-long-image.py --posts-file posts.json -o output.jpg # 自定义宽度和画质 python3 export-long-image.py --posts-file posts.json \ --width 1080 --quality 95 -o hd-output.jpg

底层用 Pillow 实现中文排版，自动查找系统中文字体（STHeiti、Hiragino、NotoSansCJK），支持自动换行。依赖 Python 3.10+ 和 Pillow。

7
👤 用户分析 — user_profile
▼

获取任意用户的主页信息和笔记列表，用于竞品分析和达人研究。

# 获取用户主页（user_id 和 xsec_token 从搜索结果中提取） user-profile.sh "user_id" "xsec_token"

返回用户基本信息和该用户发布的笔记列表。可以配合 search_feeds 批量分析某个领域的头部创作者。

8
🔐 登录与会话管理
▼

基于 Cookies 的会话管理，支持多种 Cookies 来源路径和自动同步。

# 启动服务 start-mcp.sh # 检查登录状态 status.sh # 首次登录：获取二维码 mcp-call.sh get_login_qrcode # 返回 Base64 PNG # 重置登录 mcp-call.sh delete_cookies stop-mcp.sh && start-mcp.sh

Cookies 有效期约 30 天，存储在 /tmp/cookies.json。支持从 ~/cookies.json、~/.xiaohongshu/cookies.json 或 XHS_COOKIES_SRC 环境变量自动同步。Linux 无桌面环境下会自动启动 Xvfb 虚拟显示。

值得关注，但有门槛

👍 优点

  • 工程化程度在 Skill 市场里属于顶流。15 个脚本文件，覆盖搜索、详情、评论、发布、用户分析、热点追踪、长图导出——几乎把小红书的主要操作都 API 化了。不是空壳 Prompt，而是有真正可跑的代码。
  • 架构设计干净。mcp-call.sh 统一调度层 + 各功能脚本分工明确，很"Unix 味"。JSON-RPC 协议 + Session ID 管理，安全性考虑到位（shell 注入防护、cookies 权限 600）。
  • track-topic 是杀手级功能。一行命令完成"搜索→拉详情→分析评论→生成报告"的全链路，还能导出飞书。对内容运营来说，这个功能单独就值回票价。
  • Linux 服务器适配细致。自动检测桌面环境、自动启动 Xvfb、自动选择可用 display 号、清理僵尸锁文件……这些细节说明作者真的在生产环境跑过。
  • SKILL.md 文档质量高。数据流图、参数说明、脚本对照表一应俱全，中文文档对中文工具做到了本土化。

👎 缺点

  • 安装门槛高，非技术用户劝退。需要安装 Go 编写的 xiaohongshu-mcp 二进制、jq 命令行工具、Python 3.10+、Pillow……这不是"点一下就能用"的东西。对于小红书的主要用户群体（内容创作者、运营），这个技术门槛过高。
  • 登录机制脆弱。基于 Cookies 的登录有效期只有 30 天，过期需要重新扫码。这意味着你不能真正实现"无人值守"的自动化——每个月至少要人工介入一次。而且同一账号不能多客户端同时操作，限制了团队协作。
  • 发布限制多，自动化空间有限。标题 ≤ 20 字符、正文 ≤ 1000 字符、日发布 ≤ 50 条——这些限制不是 Skill 的问题，是小红书平台的限制。但 Skill 没有提供任何"规避策略"或"最佳实践"建议，用户可能踩到频率限制被封号。
  • track-topic.py 的"分析"太粗糙。号称"评论关键词分析"和"趋势分析"，实际上分析部分就硬编码了几行"用户对该话题的关注度较高"、"评论区互动活跃"。没有真正的 NLP 分析，也没有调用 LLM 做智能总结。这是最大的遗憾。
  • Windows 兼容性为零。所有脚本都是 .sh（Bash），依赖 curl/jq/nohup 等 Unix 工具。Windows 用户需要 WSL 才能跑，但文档里没提这回事。

💡 谁最该用？

有技术背景的内容运营——如果你同时懂一点命令行，又需要做小红书竞品分析和热点追踪，这个 Skill 能让你效率翻倍。把 track-topic 设成定时任务，每天自动拉取某个关键词的热帖报告。

小红书工具开发者——这个 Skill 的架构设计是很好的参考。MCP 协议 + Shell 脚本的分层思路、JSON-RPC 的会话管理、Cookies 同步机制，都值得借鉴。

纯内容创作者——先别碰。等有人基于这个 Skill 做出"一键操作"的 GUI 工具再说。或者，让你的 AI 助手帮你调用——这才是它作为 Skill 的正确打开方式：你不需要自己跑命令，让 AI 替你跑。
